# useDefineForClassFields 详解

## 什么是 useDefineForClassFields

`useDefineForClassFields` 是 TypeScript 编译器选项之一，控制类字段（class fields）的编译输出方式。它决定了 TypeScript 是使用 `Object.defineProperty` 还是简单赋值来初始化类的实例属性。

这个选项的出现源于 TC39 对类字段语义的标准化。在 TC39 规范最终确定之前，TypeScript 采用了自己的类字段初始化方式（简单赋值）。当 TC39 规范确定使用 `[[Define]]` 语义后，TypeScript 引入了这个选项来对齐标准行为。

## useDefineForClassFields 的默认值

`useDefineForClassFields` 的默认值取决于 `target` 配置：

| target 值 | useDefineForClassFields 默认值 |
|-----------|-------------------------------|
| `ES2022` 及以上 | `true` |
| `ESNext` | `true` |
| `ES2021` 及以下 | `false` |

这是因为 `[[Define]]` 语义是在 ES2022 中正式纳入规范的。当 `target` 为 `ES2022+` 时，TypeScript 默认对齐标准行为。

## true 和 false 的不同表现

### 示例代码

```typescript
class Animal {
  name = 'animal';
}

class Dog extends Animal {
  name = 'dog';
}
```

### useDefineForClassFields: false（赋值语义）

编译输出等价于：

```javascript
class Animal {
  constructor() {
    // 简单赋值，走原型链的 setter
    this.name = 'animal';
  }
}

class Dog extends Animal {
  constructor() {
    super();
    // 简单赋值，走原型链的 setter
    this.name = 'dog';
  }
}
```

关键特征：
- 类字段通过 `this.xxx = value` 赋值
- 会触发原型链上的 setter
- 子类字段赋值发生在 `super()` 之后
- 未赋初始值的字段（如 `name!: string`）不会生成任何赋值代码

### useDefineForClassFields: true（Define 语义）

编译输出等价于：

```javascript
class Animal {
  constructor() {
    // 使用 Object.defineProperty，不走原型链的 setter
    Object.defineProperty(this, 'name', {
      configurable: true,
      enumerable: true,
      writable: true,
      value: 'animal'
    });
  }
}

class Dog extends Animal {
  constructor() {
    super();
    // 使用 Object.defineProperty
    Object.defineProperty(this, 'name', {
      configurable: true,
      enumerable: true,
      writable: true,
      value: 'dog'
    });
  }
}
```

关键特征：
- 类字段通过 `Object.defineProperty` 定义
- 不会触发原型链上的 setter
- 未赋初始值的字段（如 `name!: string`）会被定义为 `undefined`

### 核心差异对比


| 特性 | `false`（赋值语义） | `true`（Define 语义） |
|------|---------------------|----------------------|
| 初始化方式 | `this.x = value` | `Object.defineProperty(this, 'x', ...)` |
| 是否触发 setter | 是 | 否 |
| 未初始化的声明字段 | 不生成代码 | 定义为 `undefined` |
| `declare` 字段 | 不生成代码 | 不生成代码 |
| 与 TC39 标准一致 | 否 | 是 |

> 注意：`declare` 关键字在两种模式下都不会生成运行时代码，它仅用于类型声明。这是 TypeScript 提供的一种"逃生舱"，让开发者在 `useDefineForClassFields: true` 时仍能声明不需要运行时初始化的字段。

## 对装饰器功能的影响

### Stage 1 装饰器（experimentalDecorators: true）

Stage 1 装饰器（也称为 TypeScript 实验性装饰器）支持以下装饰器类型：
- 类装饰器
- 方法装饰器
- 访问器装饰器
- 属性装饰器
- **参数装饰器**（Stage 3 不支持）

#### useDefineForClassFields: false 时

这是 Stage 1 装饰器的推荐配置。属性装饰器通过 `Object.defineProperty` 在原型上定义拦截器，而 `false` 模式下类字段通过 `this.x = value` 赋值，会触发原型上的 setter，装饰器可以正常拦截属性的读写操作。

```typescript
// Stage 1 属性装饰器示例
function Log(target: any, propertyKey: string) {
  let value: any;
  Object.defineProperty(target, propertyKey, {
    get() { return value; },
    set(newVal) {
      console.log(`${propertyKey} changed to ${newVal}`);
      value = newVal;
    }
  });
}

class MyClass {
  @Log
  name = 'hello'; // false 模式：this.name = 'hello' → 触发 setter → 正常工作
}
```

#### useDefineForClassFields: true 时

属性装饰器可能失效。因为 `true` 模式下类字段通过 `Object.defineProperty` 直接在实例上定义属性，这会**覆盖**原型上装饰器定义的 getter/setter，导致装饰器的拦截逻辑被绕过。

```typescript
class MyClass {
  @Log
  name = 'hello';
  // true 模式：Object.defineProperty(this, 'name', { value: 'hello', ... })
  // 直接在实例上定义了 name 属性，覆盖了原型上 @Log 定义的 getter/setter
  // 装饰器失效！
}
```

> 这就是为什么使用 Stage 1 装饰器时，TypeScript 官方建议将 `useDefineForClassFields` 设置为 `false`。

### Stage 3 装饰器（TC39 标准装饰器）

Stage 3 装饰器是 TC39 标准化的装饰器提案，TypeScript 5.0+ 原生支持（不需要 `experimentalDecorators` 选项）。

Stage 3 装饰器支持的类型：
- 类装饰器
- 方法装饰器
- 访问器装饰器（`accessor` 关键字）
- 字段装饰器（Field Decorator）
- **不支持参数装饰器**

#### Stage 3 字段装饰器的工作机制

Stage 3 字段装饰器的签名为：

```typescript
type ClassFieldDecorator = (
  value: undefined,
  context: ClassFieldDecoratorContext
) => ((initialValue: unknown) => unknown) | void;
```

关键 API：
- `context.addInitializer(fn)`：注册一个回调函数，在类实例化时执行
- 装饰器可以返回一个函数，用于转换字段的初始值

#### useDefineForClassFields: true 时

这是 Stage 3 装饰器的推荐配置，也是与 TC39 标准一致的行为。

`addInitializer` 注册的回调在构造函数执行期间被调用，此时实例已经创建但字段尚未全部初始化。字段装饰器返回的初始值转换函数会在字段初始化时执行。

```typescript
// Stage 3 字段装饰器
function Inject(token: any) {
  return function (_value: undefined, context: ClassFieldDecoratorContext) {
    context.addInitializer(function (this: any) {
      // this 指向正在构造的实例
      // this.constructor 指向类的构造函数
      const Ctor = this.constructor;
      // 在这里注册元数据...
    });
  };
}

class A {
  @Inject(B)
  b!: B;
  // true 模式：Object.defineProperty(this, 'b', { value: undefined, ... })
  // addInitializer 回调在此之前或期间执行
}
```

#### useDefineForClassFields: false 时

`addInitializer` 回调仍然会执行，但字段的初始化行为发生了变化：

- 有初始值的字段：`this.b = value`（赋值语义）
- **无初始值的声明字段（如 `b!: B`）：不生成任何代码**

这意味着 `b!: B` 这样的声明在 `false` 模式下完全不会在运行时产生任何效果——既不会赋值，也不会定义属性。字段仅存在于类型层面。

## 对本项目（@kaokei/di）的影响分析

### 项目背景

本项目是一个基于 Stage 3 装饰器的依赖注入库。核心装饰器包括：

- `@Inject(token)`：字段装饰器，标记需要注入的依赖
- `@Self()`、`@SkipSelf()`、`@Optional()`：字段装饰器，控制依赖解析策略
- `@PostConstruct()`：方法装饰器，标记初始化方法
- `@PreDestroy()`：方法装饰器，标记销毁方法

### 装饰器元数据注册机制

本项目的字段装饰器通过 `context.addInitializer` 注册回调，在实例化时将元数据写入 `CacheMap`（基于 WeakMap）：

```typescript
function createDecorator(decoratorKey: string, defaultValue?: any) {
  return function (decoratorValue?: any) {
    return function (_value: undefined, context: ClassFieldDecoratorContext) {
      context.addInitializer(function (this: any) {
        const Ctor = this.constructor as Newable;
        const propertyName = context.name as string;
        // 将装饰器元数据存储到 CacheMap
        const ownPropertiesMetadata =
          getOwnMetadata(KEYS.INJECTED_PROPS, Ctor) || {};
        const propertyMetadata = ownPropertiesMetadata[propertyName] || {};
        propertyMetadata[decoratorKey] = decoratorValue === void 0 ? defaultValue : decoratorValue;
        ownPropertiesMetadata[propertyName] = propertyMetadata;
        defineMetadata(KEYS.INJECTED_PROPS, ownPropertiesMetadata, Ctor);
      });
    };
  };
}
```

### 依赖注入的执行流程

`Binding._resolveInstanceValue` 的执行顺序：

1. `new ClassName()` → 创建实例，触发 `addInitializer` 回调，注册元数据
2. `activate(inst)` → 执行激活钩子
3. `_getInjectProperties(options)` → 读取元数据，从容器解析依赖
4. `_injectProperties(properties)` → 通过 `Object.assign` 将依赖注入到实例上
5. `_postConstruct()` → 执行生命周期钩子

### 为什么 useDefineForClassFields: false 会导致测试失败


当前项目的 `tsconfig.app.json` 和 `tsconfig.vitest.json` 中 `target` 设置为 `ES2015`。在 `ES2015` 下，`useDefineForClassFields` 的默认值为 `false`。但项目原本显式设置为 `true`（或依赖默认值 `true`），所有测试通过。当手动设置为 `false` 后，大量测试失败。

根本原因在于 **`Object.assign` 无法覆盖 `useDefineForClassFields: false` 模式下类自身初始化的属性值**——不，准确地说，问题出在执行时序和字段初始化的语义差异上。

让我们以一个典型的测试用例来分析：

```typescript
class A {
  public name = 'A';
  public id = 1;

  @Inject(B)
  public b!: B;
}
```

#### useDefineForClassFields: true 时（测试通过）

编译后的类字段初始化使用 `Object.defineProperty` 语义。执行 `new A()` 时：

1. 构造函数执行，`addInitializer` 回调被触发，元数据 `{ b: { inject: B } }` 被注册到 CacheMap
2. `Object.defineProperty(this, 'name', { value: 'A', ... })` — name 被定义为 `'A'`
3. `Object.defineProperty(this, 'id', { value: 1, ... })` — id 被定义为 `1`
4. `Object.defineProperty(this, 'b', { value: undefined, ... })` — b 被定义为 `undefined`

然后容器执行 `_injectProperties`：

5. `Object.assign(instance, { b: resolvedB })` — b 被赋值为解析后的 B 实例

最终结果：`a.name === 'A'`，`a.id === 1`，`a.b` 是 B 的实例。✅ 测试通过。

#### useDefineForClassFields: false 时（测试失败）

编译后的类字段初始化使用赋值语义。但关键区别在于：**`b!: B` 这样没有初始值的声明字段，在 `false` 模式下不会生成任何运行时代码**。

然而，这并不是导致测试失败的直接原因。真正的问题更加微妙：

**问题的核心：`addInitializer` 回调的执行时机**

在 Stage 3 装饰器规范中，`addInitializer` 注册的回调在实例化过程中执行。但具体的执行时机与字段初始化的顺序关系，在 `useDefineForClassFields` 不同设置下可能产生差异。

当 `useDefineForClassFields: false` 时，TypeScript 编译器对类字段的处理方式发生了根本性变化。虽然 Stage 3 装饰器的 `addInitializer` 机制本身不受影响（回调仍然会执行），但编译器生成的代码结构不同，可能导致以下问题：

**问题一：字段初始化覆盖注入值**

```typescript
class A {
  public name = 'A';
  public id = 1;

  @Inject(B)
  public b!: B;
}
```

在 `useDefineForClassFields: true` 时，`b!: B` 会通过 `Object.defineProperty` 定义为 `undefined`。之后 `Object.assign` 可以正常覆盖这个值。

在 `useDefineForClassFields: false` 时，`b!: B` 不生成任何运行时代码。看起来这应该没问题——`Object.assign` 仍然可以设置 `b` 属性。但问题在于，如果编译器在 `false` 模式下改变了装饰器相关代码的生成方式，可能导致 `addInitializer` 的执行时机或行为发生变化。

**问题二：有初始值的字段覆盖注入值**

这是更关键的问题。考虑以下场景：

```typescript
class A {
  @Inject(B)
  public b!: B;

  public name = 'A';
}
```

容器的注入流程是：
1. `new A()` → 触发 `addInitializer`，注册元数据
2. 容器读取元数据，解析依赖
3. `Object.assign(instance, { b: resolvedB })` → 注入依赖

但在 `useDefineForClassFields: false` 模式下，如果编译器将字段初始化代码放在构造函数中，而 `addInitializer` 的执行时机与字段初始化交织在一起，就可能出现时序问题。

**问题三：编译器对装饰器字段的特殊处理**

在 `useDefineForClassFields: true` 时，TypeScript/esbuild 编译器知道需要按照 TC39 Stage 3 规范处理装饰器。字段装饰器的 `addInitializer` 回调在正确的时机执行。

在 `useDefineForClassFields: false` 时，编译器切换到旧的赋值语义。这可能与 Stage 3 装饰器的运行时语义产生冲突，因为 Stage 3 装饰器规范是基于 `[[Define]]` 语义设计的。**Stage 3 装饰器天然假设类字段使用 `[[Define]]` 语义**，当编译器强制使用赋值语义时，装饰器的运行时行为可能不符合预期。

### 具体失败场景分析

以测试文件 `tests/feature/API_BIND.spec.ts` 中的典型用例为例：

```typescript
class A implements IA {
  public name = 'A';
  public id = 1;

  @Inject(new LazyToken(() => B))
  public b!: IB;
}

// 测试期望：
const a = container.get(A);
expect(a.b).toBeInstanceOf(B);  // 期望 b 是 B 的实例
expect(a.b.id).toBe(2);
expect(a.b.name).toBe('B');
```

当 `useDefineForClassFields: true` 时：
- `new A()` 执行后，`a.b` 被 `Object.defineProperty` 定义为 `undefined`
- `addInitializer` 正常注册元数据
- 容器通过 `Object.assign` 将解析后的 B 实例注入到 `a.b`
- 测试通过 ✅

当 `useDefineForClassFields: false` 时：
- 编译器生成的代码与 Stage 3 装饰器的 `[[Define]]` 语义不匹配
- 可能导致 `addInitializer` 回调未正确执行，或元数据未正确注册
- 容器无法读取到注入元数据，`b` 属性未被注入
- `a.b` 为 `undefined`，测试失败 ❌

### 根本原因总结

**Stage 3 装饰器是基于 `[[Define]]` 语义设计的。** TC39 在制定装饰器规范时，类字段的 `[[Define]]` 语义已经确定（ES2022）。因此，Stage 3 装饰器的所有运行时行为——包括 `addInitializer` 的执行时机、字段装饰器的初始值转换等——都假设类字段使用 `[[Define]]` 语义。

当 `useDefineForClassFields` 设置为 `false` 时，TypeScript 编译器将类字段回退到旧的赋值语义，这与 Stage 3 装饰器的设计假设产生了根本性冲突：

1. **语义不匹配**：Stage 3 装饰器期望 `[[Define]]` 语义，但编译器生成了赋值语义的代码
2. **运行时行为异常**：`addInitializer` 回调的执行时机、字段初始化的顺序等可能与规范不一致
3. **元数据注册失败**：装饰器的元数据可能未被正确注册，导致容器无法解析依赖

### 最佳实践建议

| 装饰器类型 | useDefineForClassFields 推荐值 |
|-----------|-------------------------------|
| Stage 1（experimentalDecorators） | `false` |
| Stage 3（TC39 标准） | `true` |

对于本项目（使用 Stage 3 装饰器）：

- **必须**将 `useDefineForClassFields` 设置为 `true`（或在 `target >= ES2022` 时使用默认值）
- 如果 `target` 设置为 `ES2015`（低于 ES2022），则必须**显式**设置 `useDefineForClassFields: true`
- 不要将 `useDefineForClassFields` 设置为 `false`，否则会破坏 Stage 3 装饰器的运行时语义

### 配置示例

```jsonc
// tsconfig.app.json — 正确配置
{
  "compilerOptions": {
    "target": "ES2015",
    "useDefineForClassFields": true,  // 必须显式设置为 true
    // ...
  }
}

// tsconfig.vitest.json — 正确配置
{
  "compilerOptions": {
    "target": "ES2015",
    "useDefineForClassFields": true,  // 必须显式设置为 true
    // ...
  }
}
```

如果将 `target` 升级到 `ES2022` 或更高版本，则可以省略 `useDefineForClassFields`，因为默认值就是 `true`：

```jsonc
{
  "compilerOptions": {
    "target": "ES2022",
    // useDefineForClassFields 默认为 true，无需显式设置
  }
}
```
