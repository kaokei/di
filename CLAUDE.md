# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

`@kaokei/di` is a lightweight TypeScript dependency injection library. It uses TC39 Stage 3 decorators (no `reflect-metadata`), and supports property injection with circular dependency detection.

## Commands

```bash
# Build the library (TypeScript check + Vite bundle → dist/)
pnpm build

# Run tests (excludes tests/inversify/)
pnpm test

# Run inversify compatibility tests (uses SWC with Stage 1 decorators)
pnpm test:inversify

# Run a single test file
pnpm vitest run tests/feature/API_BIND.spec.ts

# Generate coverage report
pnpm coverage

# Docs dev server
pnpm docs:dev
```

## Architecture

### Core modules (`src/`)

| File | Role |
|------|------|
| `container.ts` | `Container` class — manages bindings, resolves tokens, supports parent/child hierarchy |
| `binding.ts` | `Binding` class — fluent API (`to`, `toSelf`, `toConstantValue`, `toDynamicValue`, `inTransientScope`), lifecycle, injection |
| `decorator.ts` | All Stage 3 decorators: `@Injectable`, `@Inject`, `@Self`, `@SkipSelf`, `@Optional`, `@PostConstruct`, `@PreDestroy`, `@LazyInject`, `@autobind`, and the `decorate()` helper |
| `token.ts` | `Token<T>` and `LazyToken<T>` — typed tokens for non-class bindings |
| `cachemap.ts` | WeakMap-based metadata store — links constructor → metadata for injected props and lifecycle hooks |
| `constants.ts` | `KEYS`, `STATUS`, `BINDING` enums, `ERRORS` messages, `hasOwn`, `isObject` |
| `interfaces.ts` | All shared TypeScript types |

### Resolution flow

1. `container.get(token)` → looks up `Binding` in `_bindings` map, walks up `parent` chain if not found
2. `Binding.get()` → dispatches to `_resolveInstanceValue`, `_resolveConstantValue`, or `_resolveDynamicValue` via static `_resolvers` map
3. `_resolveInstanceValue`: creates instance → runs activation handlers → **property injection** (`_getInjectProperties` reads `cachemap` metadata) → `@PostConstruct`
4. `@LazyInject`: defines a getter/setter on the instance at creation time; resolves on first access via `Container.getContainerOf(instance)`

### Key design decisions

- **No `reflect-metadata`**: all metadata is stored via Stage 3 `context.metadata` (WeakMap keyed by constructor) in `cachemap.ts`
- **Singleton by default**: `Binding.cache` stores the resolved value; `inTransientScope()` bypasses this
- **Circular dependency detection**: status machine (`DEFAULT` → `INITING` → `ACTIVATED`); constructor injection would throw `CircularDependencyError`, property injection resolves after the instance is cached so it's safe
- **`LazyToken`**: wraps a callback `() => CommonToken` to break circular import cycles at the module level

### Test structure (`tests/`)

Tests are organized by feature area: `feature/`, `hooks/`, `activation/`, `decorate/`, `decorator/`, `lazyinject/`, `special/`, `container/`, `errors/`, `quality/`.

The `tests/inversify/` directory tests API compatibility with InversifyJS v6 using Stage 1 decorators — run separately with `pnpm test:inversify` (uses SWC via `vite.config.inversify.ts`).

### Build output

Vite builds two formats from `src/index.ts`:
- `dist/index.js` (ESM)
- `dist/index.cjs` (CJS)

TypeScript declarations are generated by `vite-plugin-dts`; `.d.cts` files are generated via the `beforeWriteFile` hook for CJS consumers.
